DataQualityDashboard Tutorial on a Staged OMOP CDM

Introduction

This tutorial is designed to help OHDSI users understand how the DataQualityDashboard (DQD) thinks about data quality, how to inspect a finding, and how to reason about remediation. The tutorial uses a synthetic OMOP CDM built from Synthea plus a small set of injected failures across fatal, convention, and characterization.

The goal is not only to show how to run DQD, but to show how to go from:

  1. a failed check in DQD
  2. to the metadata that defines the check
  3. to the SQL template shipped in the DQD package
  4. to a readable diagnostic query you can use to investigate the affected rows
  5. to a discussion of realistic remediation paths

The tutorial still demonstrates a local fix in the working copy, but those fixes should be read as tutorial examples rather than the universal production response.

How DQD Organizes Checks

DQD evaluates the OMOP CDM through multiple lenses:

  • TABLE checks look at whether expected tables exist and whether table-level expectations are satisfied.
  • FIELD checks look at requiredness, keys, datatypes, completeness, and temporal logic at the column level.
  • CONCEPT checks look at concept-level plausibility and semantic use patterns.

DQD also labels checks by severity:

  • fatal: structural issues that can invalidate downstream analysis or indicate the CDM is not being represented safely.
  • convention: OMOP usage expectations and semantic consistency checks.
  • characterization: screening signals for plausibility and completeness patterns that deserve investigation.

The key fields you see in DQD results are intended to be interpretable:

  • checkName: the reusable check family
  • checkDescription: what DQD is asking
  • checkLevel: table, field, or concept
  • cdmTableName and cdmFieldName: where the check applies
  • numViolatedRows and pctViolatedRows: how much evidence DQD found
  • thresholdValue: the expected tolerance for failure
  • sqlFile: the package SQL template behind the check
  • category, subcategory, and context: the DQD/Kahn framing for the check

The Kahn Framework in DQD

DQD maps checks to the Kahn framework so that a finding is not just “bad data,” but a signal about the kind of quality question being raised.

  • Conformance means the data fits the expected structural or semantic representation of OMOP.
  • Completeness means the data is present to the degree expected for the domain or field.
  • Plausibility means the values or dates are believable enough to support intended use, while still allowing for legitimate real-world edge cases.

In practice:

  • a conformance finding often points to ETL logic, key management, or schema relationships
  • a completeness finding often points to missing source extraction, derivation gaps, or scoped omissions
  • a plausibility finding often points to chronology, linkage, or workflow timing problems that require interpretation

DQD is surfacing screening signals. It is not adjudicating root cause. A failed check should usually start a conversation that includes data owners, ETL maintainers, and sometimes source-system stakeholders.

Fatal Check 1: Duplicate MEASUREMENT_ID

Duplicate primary keys are a structural problem. In OMOP, record identifiers are used to anchor joins, maintain row identity, and avoid accidental duplication in downstream logic. If two different rows share the same identifier, the ambiguity may propagate into analytic datasets and audit trails.

In a production setting, the remediation usually starts upstream:

  • review source-system identifiers
  • inspect the ETL rule that generates or preserves primary keys
  • determine whether the issue is duplication, collision, or incorrect deduplication
  • rerun the ETL or controlled backfill after the root cause is understood

For this tutorial, we will still apply a local fix in the working copy, but only as one demonstration path.

Fatal Check 2: Overlapping OBSERVATION_PERIOD

Observation periods are used to define when a person is considered under observation. Overlap can distort denominators, incidence calculations, and cohort logic if the same span of time is counted inconsistently. In a real environment, overlap may reflect business rules in observation-period derivation rather than a simple one-row problem.

Potential remediation paths include:

  • review how contributing event windows are combined into observation periods
  • inspect whether overlap should be merged, split, or removed
  • verify whether payer enrollment, claims, or EHR events are being interpreted correctly
  • rerun derivation logic after the business rule is clarified

The tutorial fix deletes the injected overlapping row so the DQD feedback loop is easy to see.

Convention Check 1: Person Coverage in PROCEDURE_OCCURRENCE

Low person-level coverage in a domain can signal ETL omissions, incomplete source extracts, or business rules that excluded data unexpectedly. It is often a domain-scoped issue rather than an error on one individual row. Thresholds in DQD are expectations, not universal truths, so the right interpretation depends on the source and use case.

Potential remediation paths include:

  • review the source extract for procedure data
  • confirm whether some procedures were intentionally excluded
  • inspect mapping or ingestion rules
  • verify whether the domain should have broad person coverage in this source

The tutorial remediation restores deleted rows, but in a real environment the better fix often lives in extraction, transformation, or derivation logic.

Convention Check 2: CONDITION_ERA Completeness

Condition eras are derived constructs. Missing eras often point to a derivation pipeline issue, not simply a bad base event row. That is why this kind of finding is useful operationally: it can tell you that a downstream derived table is not behaving as expected even when the base clinical event table appears populated.

Potential remediation paths include:

  • rerun condition era derivation after source or ETL corrections
  • review the era-building logic and the assumptions behind it
  • verify whether the available condition events should generate eras in the first place

The tutorial remediation restores deleted backup rows so that you can see the DQD rerun behavior clearly.

Characterization Check 2: Start Date Before End Date

Reversed start and end dates are suspicious because they often indicate transformation mistakes, parsing errors, or date derivation problems. But again, the remediation question is broader than “patch the row.” You usually want to understand whether the underlying issue came from the source, the ETL, or date normalization logic.

Potential remediation paths include:

  • inspect source start/end fields
  • review transformations for truncation or date-order reversal
  • validate timezone or datetime-to-date conversion logic
  • rerun the ETL after the upstream logic is corrected

The tutorial fix restores backed-up dates in the working copy so that the DQD rerun is easy to interpret.

Wrap-up and Discussion

This tutorial used DQD directly, then unpacked how a failed check is described, where its SQL template lives in the package, and how to write a readable diagnostic query to investigate the affected rows.

The most important takeaways are:

  • DQD findings are screening signals, not root-cause analyses
  • the Kahn framing helps you understand what kind of quality question is being raised
  • the same failed check can imply different remediation paths depending on source context and governance
  • a local SQL fix is useful in a tutorial, but real remediation often belongs in source systems, ETL logic, or controlled data refresh processes

From here, you can compare the tutorial snapshots against the clean baseline, inspect the saved JSON results in the DQD dashboard, and adapt the same “finding -> metadata -> template -> diagnosis -> remediation options” workflow to other DQD checks.
