DataQualityDashboard Tutorial on a Single Tutorial DuckDB

Introduction

This tutorial uses a single tutorial database, data/syntheaCDMWithDq.duckdb, built from a committed raw Synthea export with curated data quality issues. The same DuckDB file contains both the transformed OMOP CDM schema, cdm, and the raw staging schema, synthea_native, so students can move directly from a DQD finding to the transformed row and then back to the upstream raw record.

The tutorial keeps the same overall teaching arc:

  1. run DQD
  2. inspect the failing check metadata
  3. view the package SQL template behind the check
  4. run a readable diagnostic query
  5. trace the issue back to the raw staging schema when relevant
  6. apply a local demonstration fix in a temporary working copy
  7. rerun DQD to confirm the effect

Tutorial Assets

The key assets in this workflow are:

  • clean raw source files in data/syntheaRaw/
  • committed mutated raw source files in data/syntheaRawWithDqIssues/
  • the single tutorial database data/syntheaCDMWithDq.duckdb

Inside syntheaCDMWithDq.duckdb:

  • cdm holds the OMOP CDM tables DQD evaluates
  • synthea_native holds the staged raw Synthea tables loaded by the ETL
  • main holds tutorial metadata tables that track the curated issues and fixes

/*********
Table Level:  
MEASURE_OBSERVATION_PERIOD_OVERLAP
Determine what #/% of persons have overlapping or back-to-back observation periods

Parameters used in this template:
schema = @schema
cdmTableName = @cdmTableName
{@cohort & '@runForCohort' == 'Yes'}?{

}

SELECT 
    num_violated_rows, 
    CASE 
        WHEN denominator.num_rows = 0 THEN 0 
        ELSE 1.0*num_violated_rows/denominator.num_rows 
    END AS pct_violated_rows, 
    denominator.num_rows AS num_denominator_rows
FROM
(
    SELECT 
        COUNT_BIG(violated_rows.person_id) AS num_violated_rows
    FROM
    (
        /*violatedRowsBegin*/
        SELECT DISTINCT
            cdmTable.person_id 
        FROM @cdmDatabaseSchema.observation_period cdmTable
        {@cohort & '@runForCohort' == 'Yes'}?{
            JOIN @cohortDatabaseSchema.@cohortTableName c 
                ON cdmTable.person_id = c.subject_id
                AND c.cohort_definition_id = @cohortDefinitionId
        }
        JOIN @cdmDatabaseSchema.observation_period cdmTable2 
            ON cdmTable.person_id = cdmTable2.person_id
            AND cdmTable.observation_period_id != cdmTable2.observation_period_id
        WHERE (cdmTable.observation_period_start_date <= cdmTable2.observation_period_end_date 
            AND cdmTable.observation_period_end_date >= cdmTable2.observation_period_start_date)
            OR (DATEADD(day, 1, cdmTable.observation_period_end_date) = cdmTable2.observation_period_start_date)
            OR (DATEADD(day, 1, cdmTable2.observation_period_end_date) = cdmTable.observation_period_start_date)
        /*violatedRowsEnd*/
    ) violated_rows
) violated_row_count,
(

Convention Check 1: fkDomain on MEASUREMENT.MEASUREMENT_CONCEPT_ID

This failure starts from marked rows in data/syntheaRawWithDqIssues/observations.csv. The raw marker identifies the visit in synthea_native.observations, and the tutorial then adjusts one measurement row from that same visit to an Observation concept so DQD flags MEASUREMENT.MEASUREMENT_CONCEPT_ID for fkDomain.

Wrap-up

The single-DB workflow makes the teaching loop tighter:

  • DQD evaluates cdm
  • the same DuckDB file keeps the raw staging tables in synthea_native
  • main stores tutorial lineage and fix metadata

That lets students move from a failed check to the transformed row, then to the staged raw record, without switching databases or relying on prebuilt severity snapshots.
